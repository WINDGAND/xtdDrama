## 模块一：用户洞察与问题定义

### 1. 目标用户

- **核心画像**：18-24 岁的 Z 世代大学生群体及初入职场的年轻白领  
- **身份特征**：数字原住民，各大社交平台（QQ、微信、小红书、抖音）的重度活跃者  
- **行为习惯**：碎片化表达、热衷 Meme（梗图）文化、表情包斗图、“发疯文学”和抽象文化；既是内容消费者，也是渴望互动、需要被看见与认同的“表达者”  
- **心理特征**：在“学业/工作高压”与“现实生活平庸”夹击下，精神状态游走在“佛系”与“发疯”之间，迫切需要情绪释放出口  

### 2. 用户痛点

传统社交图文分享正在面临“社交表达通缩”。我们挖掘出三大真实痛点（非伪需求）：

- **痛点一：精装朋友圈带来的分享焦虑（素材门槛高）**  
  年轻人渴望分享以获取社交货币（点赞、评论），但现实常是“毛坯人生”。日常平庸瞬间缺乏传播性，用户“想发却无图可发”，大量情绪被压在草稿箱。

- **痛点二：负面情绪缺乏幽默的宣泄出口（情绪痛点）**  
  面对 DDL、导师批评、错过公交等“小确丧”，直接文字抱怨显负能量、影响社交人设。用户需要“自嘲式、幽默化、夸张化”的视觉表达，把悲剧变喜剧，换来共鸣与安慰。

- **痛点三：发布后的社交空窗期折磨（反馈痛点）**  
  动态发布后用户反复刷新期待反馈，长时间无人问津会产生强烈挫败感。平台缺乏即时的“电子情绪价值反馈”机制。

### 3. 使用场景

小题大Drama 以“小题大作”的理念，专攻高频痛点场景：

#### 场景一：【高压下的精神状态外化（DDL受难记）】

- **时间/地点**：凌晨 1 点，图书馆或宿舍机位前  
- **事件**：屏幕上密密麻麻的论文修改意见和工作群消息，用户崩溃  
- **互动**：用户拍下满屏 DDL + 憔悴的脸发给 小题大Drama，期待获得一张能精准表达“想死但还得肝”的夸张神图（例如：屏幕伸出克苏鲁触手缠绕自己），用于朋友圈自嘲  

#### 场景二：【平庸/倒霉瞬间的戏剧化重构】

- **时间/地点**：午餐时间的食堂  
- **事件**：手抖把刚打好的红烧肉掉在桌面上  
- **互动**：拍下红烧肉，AI 识别后将其重构为“末日陨石坠落地球”的史诗灾难片海报，把倒霉瞬间变成社交圈爆款梗图

## 模块二：产品方案设计

### 1. 产品概述

一句话说明白：**小题大Drama 是一款基于多模态大模型和可控图像生成的「AI情绪重构与社交分发」产品。**

它做三件事：

- **看懂你的无聊日常**
- **用极度夸张的 AI 视觉手法重构现实**（生成 Live 图/动图）
- **一键发布并提供 AI 互动（点赞 + 评论）**的情绪价值闭环（25 秒内陆续出现）

### 1.1 评委偏好维度（强约束：需求与验收口径）

> 本小节同步了腾讯官方评委曝出的喜爱维度。后续新增功能、改交互、改模型策略、改 prompt/模板，都必须逐条对齐；若不满足，应视为“不通过验收”。

1) **真实痛点（避免伪需求）**  
从用户角度出发，优先解决“分享焦虑 / 情绪宣泄 / 发布后社交空窗期”等高频真实问题；避免为了展示能力而增加无意义步骤或功能堆叠。

2) **可真实落地（避免天马行空）**  
所有核心链路应可在现有工程形态（Serverless + Supabase + TokenHub）下闭环跑通：可部署、可调用、可存储、可回放、可复现。对外部依赖（模型/存储/带宽/权限）要有降级与失败反馈。

3) **AI 原生（拿掉 AI 就难以正常使用）**  
产品价值必须由 AI 核心能力驱动：多模态感知（看懂图）、决策收敛（Guess&Refine）、可控生成（结构一致性重绘）、拟真互动（NPC 评论/点赞）。若移除 AI，这条链路应无法以传统手段等价实现。

4) **避免技术自嗨**  
技术选择与模型参数应服务体验，不以“更复杂/更先进”为目标；任何新增的模型、链路、参数、组件，都必须能明确带来用户可感知的收益（更快、更稳、更像真人、更好玩）。

5) **趣味性与创新性兼具**  
趣味来自“现实瞬间 → 戏剧化重构 → 社交反馈”的反差与共鸣；创新来自“免 prompt 的 Guess&Refine + 结构一致性重绘 + NPC 社交闭环”的组合，而不是单点炫技。

6) **赛题④难点：风格控制与去 AI 感（提示词/策略）**  
需要稳定控制图片/视频/链接等输入输出的风格一致性，避免“生硬的 AI 味”。策略方向包括但不限于：  
- 通过结构约束（参考图/边缘/景深等）保持空间一致性  
- 通过系统级 Prompt 模板与负向约束保持风格稳定  
- 通过“可选项按钮”替代自由输入 prompt，降低语义漂移与随机性  
- 通过语言与互动人设约束，使文案更像真人表达（克制、不油、不夸夸群）

### 2. 核心功能（解决核心痛点的抓手）

- **万物皆可 Drama（多模态感知输入）**  
  支持用户发送任意场景图片/短视频，AI 提取实体、场景关系与用户潜在情绪状态。

- **Guess & Refine 情绪互动罗盘**  
  告别冷冰冰 Prompt 输入框。AI 像“捧哏”一样主动调侃，并给出 3 个极具网感的“发疯风格选项”（如：赛博牛马风、克苏鲁吞噬风、吉卜力治愈风）供一键生成，降低思考门槛。

- **空间一致性重绘（ControlNet 引擎）**  
  核心壁垒：生成图与原图在物理轮廓与透视空间上保持高度一致。红烧肉仍在原位置，只是变成陨石，保留“梗”的反差灵魂。

- **AI 赛博社交场（NPC Agent）**  
  一键分享至内部 Drama 广场。发布后，5 位 AI 互动角色（Emma、Liam、Olivia、Noah、Sophia）会在 **25 秒内陆续点赞与评论**，形成“电子情绪价值反馈”，消灭发布后的社交空窗焦虑。  
  - **风格要求**：更像真实的人类表达，不过度浮夸；每位角色有稳定但克制的表达偏好（语气、关注点、用词习惯不同）。

### 3. 产品架构与逻辑流程图（感知-决策-生成 任务拆解）

> 注：在 PDF 中可配对应思维导图/流程图。此处给出逻辑描述，直接回应赛题 3 和赛题 4。

整体遵循三层架构：**感知（Perception）- 决策（Decision）- 生成与反馈（Generation & Feedback）**

#### 环节一：用户发图与【感知层】（意图拆解）

- 用户上传真实生活照片（可选一句话吐槽）
- 前端通过 API 将图像及文本推流至 **腾讯混元多模态大模型（Vision LLM）**
- 大模型输出结构化 JSON，例如：

```json
{
  "核心实体": "电脑屏幕及密集文字",
  "环境": "昏暗书桌",
  "情感标签": "焦虑/高压"
}
```

#### 环节二：AI 互动与【决策层】（Guess & Refine）

- 中枢调度 Agent 接收情感标签，从云端策略库匹配“高压情绪化解策略”
- 界面弹出破冰回复，例如：“被DDL冲烂了吧哥们？选个姿势发疯：”
- 并发呈现 3 个风格按钮（预埋不同系统级 Prompt 模板），用户点击完成意图收敛与确认（如选择“克苏鲁吞噬风”）

#### 环节三：AI 回图与【生成分发层】（空间控制与渲染）

- 调用 **ControlNet（Depth/Canny）** 提取边缘与景深信息
- 结合用户选择风格 Prompt，送入大参数绘图模型（如 SDXL / 腾讯云 TI 生图 API）重绘
- 回传最终图片/Live 图，用户可一键发布至广场
- 后台唤醒 NPC Agent 评论组：根据生成图 + 配文生成符合人设的评论，完成情绪闭环

### 4. 创新与差异化

相较于美图秀秀滤镜、妙鸭相机或通用 Midjourney：

- **不仅是工具，更是社交应用**  
  不止“图下发到本地”，而是延伸到“内容生产 -> 社交分发 -> 情绪满足”的闭环：NPC 评论区补足情绪价值与互动反馈。

- **免 Prompt 的交互革命**  
  采用“原图 + AI 猜测 + 用户多选一（Guess & Refine）”范式，大幅降低非技术用户门槛。

- **保留物理真实的造梗机器**  
  严格锁定图像结构，仅做戏剧性材质与氛围替换，制造高反差梗图更稳定、更易传播。

## UI/UX 视觉与交互规范（Apple Style · 偏 macOS）

> 本章节是**后续所有前端迭代的强制设计约束**。任何新增页面/组件/动效，必须以此为准，确保风格一贯。

### 1. 设计哲学（必须遵守）

- **内容优先**：隐去工具痕迹，只留内容本身。避免“炫技式视觉”，信息层级清晰、可读性第一。
- **极致克制**：不用霓虹、发光、强渐变、赛博朋克未来风；避免高饱和+高对比造成的视觉疲劳。
- **精密秩序**：通过网格、留白、字体层级与细分隔线建立秩序，而非依赖装饰性效果。
- **材质呼吸感**：用轻微层级、细腻阴影、柔和灰阶表达“材质”，动效短促、安静、不抢内容。

### 2. 主题策略（必须遵守）

- **默认浅色为主**：全程以浅色体验为主，不允许因业务流程（如 Guess 成功）自动切换主题。
- **暗色模式可选**：仅允许“跟随系统”或“用户手动切换”，暗色风格为 **macOS 深色**（低饱和、低对比、无霓虹）。

### 3. 颜色体系（强约束）

- **唯一高亮色：系统蓝（Apple Blue）**  
  用途：主 CTA、可点击文本、聚焦态（focus ring）、选中态提示。  
  禁止：紫粉霓虹、发光描边、渐变背景作为主按钮。
- **灰阶为主**：背景/卡片/文字/分隔线均应使用中性灰阶，避免纯黑纯白硬对比。
- **语义色**：成功/失败/警告仅用于状态反馈（如任务完成/失败），面积要小、对比要克制。

### 4. 排版与信息层级（强约束）

- **标题**：字重偏中（semibold），避免过粗“海报感”。行高紧凑但不压迫，确保可读。
- **正文**：以清晰为先，减少花哨修饰；辅助信息使用更浅灰与更小字号。
- **数字/模型/ID**：使用等宽字体（mono），并避免过度强调。

### 5. 间距与布局（强约束）

- **留白是结构的一部分**：通过 padding/margin 建立节奏，不用装饰性线条堆砌。
- **不锁死最大宽度**：页面可随屏幕变宽而扩展，但**阅读内容区域**应保持合理行宽与栅格秩序（避免正文无限拉长）。
- **对齐与网格**：所有组件在同一列宽、同一基线对齐；边距要一致，不出现“随手堆叠”的凌乱感。

### 6. 组件外观规范（强约束）

- **非卡片化优先（强约束）**：本项目默认采用「朋友圈式信息流」的**平铺内容**表达方式，通过**留白 + 分隔线 + 字体层级**建立秩序，避免用大块 Card/Panel 把内容“框起来”。  
  - **允许的容器场景（例外）**：仅在“设置项分组 / 表单区块 / 强状态提示（错误/空态）/ 图片上传区”这类需要明确边界的区域，才允许使用轻量容器。  
  - **容器形态约束**：即使使用容器，也必须极轻（细边/弱底/无重阴影），不可出现“厚卡片感”和强分层。
- **按钮**：\n+  - 主按钮：系统蓝底白字，圆角中等（偏 macOS 面板），无渐变无发光。\n+  - 次按钮：浅灰背景/细边，hover 仅做轻微色阶变化。\n+- **选项选择（Option/Segmented）**：选中态使用浅蓝背景 + 细边或指示点；不要霓虹描边。
- **输入/上传区域**：强调“清晰可理解”，状态反馈（hover/dragging/error）差异要细腻但明确。

### 7. 动效与反馈（强约束）

- **动效短促**：以 150–250ms 的淡入/位移为主，避免弹跳/强 spring 影响克制感。
- **不做“赛博”特效**：禁止发光、粒子、霓虹闪烁、过度 blur 毛玻璃作为主视觉。
- **加载与等待**：信息要明确但不喧闹。允许极简呼吸点/文字渐隐渐显，避免大面积动画。

### 8. 禁止清单（直接拒绝）

- 霓虹渐变文字、霓虹发光阴影、赛博紫粉渐变 CTA
- 以“黑底+霓虹”为主导的视觉叙事（除非明确改 PRD 并说明原因）
- 过度毛玻璃（大面积 blur 覆盖内容导致可读性下降）
- 夸张动效（弹跳、强缩放、闪烁、复杂粒子）

## 模块三：AI 原生能力说明

### 1. AI 核心能力应用

小题大Drama 是一个 AI-Native 产品，深度融合：

- **多模态理解（Vision LLM）**：产品“眼睛”，负责图像语义拆解与情绪识别，支撑高情商对话  
- **LLM 与角色扮演（LLM & Agent）**：产品“大脑和嘴巴”，负责策略生成、选项推荐与 NPC 人设一致互动  
- **可控生成式 CV（SDXL + ControlNet）**：产品“画笔”，在约束下实现像素级结构一致的风格化重绘  

### 2. AI 如何解决痛点（去掉AI产品还能成立吗？）

**结论：去掉 AI，本产品将彻底崩溃，无法成立。**

- **缺乏 AI 视觉理解**：传统代码无法理解“红烧肉掉地上”背后的“沮丧”情绪，无法做语境化选项猜测  
- **缺乏 AI 可控生成**：滤镜只能改色调，无法把“显示器”重构为“克苏鲁触手”这种语义生成与夸张化  
- **缺乏 LLM 驱动的 Agent**：规则机器人生硬；只有 LLM 才能针对每次发布的个性图片生成懂梗、拟真的秒回互动  

### 3. AI 技术方案（应对赛题 2 的挑战）

为保证极速响应与一致体验，采用前后端分离与云原生 Serverless：

- **前端交互与部署**：Next.js 构建轻量 Web App，通过 **Vercel + Cloudflare** 部署与加速（纯 Serverless，不自建后端服务）  
- **中枢逻辑与感知决策（AI 大脑）**：接入腾讯混元 API  
  - 多模态：原图分析 + 情绪提取  
  - 文本能力：Guess & Refine 选项生成 + NPC 评论生成  
- **生图推理与一致性保障（视觉核心）**  
  - **结构一致性**：ControlNet（Canny / Depth）提取骨架，约束扩散过程  
  - **风格一致性**：自研“动态系统 Prompt 注入引擎”  
    - Prompt 公式：`[核心实体] + [极致夸张场景描述词] + [Cinematic Lighting, 8k, Unreal Engine 5]`  
    - 通过锁定 Negative Prompt，确保输出稳定呈现“史诗感”或“二次元”等目标风格  
  - **云端推理**：对接腾讯云 TI 或第三方极速推理（如 Fal.ai）实现秒级下发  

#### 部署与落地架构（Vercel + Cloudflare + Supabase，无传统后端）

本项目目标是做到“拿到仓库即可部署”，因此采用 **全托管 + Serverless** 的工程形态：不运行常驻后端进程、不自建服务器，仅通过前端与云函数/边缘函数完成业务闭环。

- **Vercel（应用托管 + Serverless Functions）**  
  - 托管 Next.js 前端与路由  
  - 承载需要保密的服务端逻辑（如：调用大模型/生图接口的密钥、签名、风控校验、请求聚合）

- **Cloudflare（边缘加速/安全/缓存）**  
  - 静态资源与页面的边缘缓存、HTTPS 与基础安全防护  
  - 可选：将部分轻量逻辑下沉至边缘（如限流、AB、地理就近与缓存策略）

- **Supabase（数据库 + Auth + Storage）**  
  - **Postgres**：存储用户、作品、发布、互动等结构化数据  
  - **Auth**：邮箱/手机号/OAuth 登录与会话管理（前端可直接使用 Supabase 客户端）  
  - **Storage**：存储用户上传原图/生成结果图/录屏等文件，并通过策略控制访问权限

#### 典型请求流（从“发图”到“回图”）

1. 前端在 Vercel 上提供上传与选择风格的交互  
2. 原图可直传 Supabase Storage（或经函数中转），并写入 Postgres 记录元数据  
3. 前端调用 Vercel Serverless Function：函数读取必要信息，安全地调用多模态/生图 API  
4. 生成结果落 Supabase Storage，并回写生成记录、评论/互动记录到 Postgres  
5. 前端拉取结果与互动数据，在 Cloudflare 边缘加速下完成分发与浏览

> 说明：在“无传统后端”的约束下，所有需要私密凭证的第三方调用，都应放在 Vercel 的 Serverless Functions（或 Cloudflare Workers）中完成；前端仅拿到受控的短期凭证/公开资源地址，避免密钥泄露。
