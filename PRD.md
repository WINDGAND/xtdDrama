## 模块一：用户洞察与问题定义

### 1. 目标用户

- **核心画像**：18-24 岁的 Z 世代大学生群体及初入职场的年轻白领  
- **身份特征**：数字原住民，各大社交平台（QQ、微信、小红书、抖音）的重度活跃者  
- **行为习惯**：碎片化表达、热衷 Meme（梗图）文化、表情包斗图、“发疯文学”和抽象文化；既是内容消费者，也是渴望互动、需要被看见与认同的“表达者”  
- **心理特征**：手机相册里不缺图，缺的是"值得发"的图——有分享欲、有表达欲、热爱吐槽与自嘲，但日常素材（不论倒霉的、平凡的还是小确幸的）总差一口"有意思"的气，发与不发之间永远在纠结  

### 2. 用户痛点

传统社交图文分享正在面临“社交表达通缩”。我们挖掘出三大真实痛点（非伪需求）：

- **痛点一：精装朋友圈带来的分享焦虑（素材门槛高）**  
  年轻人渴望分享以获取社交货币（点赞、评论），但现实常是“毛坯人生”。日常平庸瞬间缺乏传播性，用户“想发却无图可发”，大量情绪被压在草稿箱。

- **痛点二：表达欲与内容趣味性之间的落差（素材无聊但想发）**  
  拍下来的图，就是一杯普通拿铁、一只懒懒的猫、一个平平无奇的傍晚——有分享的冲动，但发出去又觉得"太无聊没人理"。同样，遇上倒霉瞬间，直接抱怨显负能量；想自嘲，又不知怎么把它变好玩。两种场景的本质相同：**真实素材本身不够有趣，而年轻人缺的正是把它变有意思的那一步**。  

- **痛点三：发布后的社交空窗期折磨（反馈痛点）**  
  动态发布后用户反复刷新期待反馈，长时间无人问津会产生强烈挫败感。平台缺乏即时的“电子情绪价值反馈”机制。

### 3. 使用场景

小题大Drama 以“小题大作”的理念，专攻高频痛点场景：

#### 场景一：【高压下的精神状态外化（DDL受难记）】

- **时间/地点**：凌晨 1 点，图书馆或宿舍机位前  
- **事件**：屏幕上密密麻麻的论文修改意见和工作群消息，用户崩溃  
- **互动**：拍下书桌与屏幕，AI 感知"焦虑/疲惫"情绪标签，提供三个有反差感的改造方向（显示器边缘溢出袅袅咖啡蒸汽 / 书堆投下的阴影凝成密集倒计时数字 / 屏幕上的错误代码自行漫溢成一道像素洪水），把憔悴深夜变成值得转发的梗图  

#### 场景二：【平凡幸福瞬间的 Drama 化（小确幸也可以很有意思）】

- **时间/地点**：周末下午，家或咖啡馆  
- **事件**：随手拍了一杯自制拿铁，拉花歪七扭八，想发朋友圈又觉得"太普通了没意思"  
- **互动**：上传图片，AI 识别"日常/小确幸"情绪，提供三个方向（精品咖啡馆大片风 / 复古电影海报风 / 宫崎骏治愈插画风），保留杯子物理位置与透视，背景与光线重绘，"歪拉花"变成"值得 Po 的精致下午茶"

## 模块二：产品方案设计

### 1. 产品概述

一句话说明白：**小题大Drama 是一款让年轻人「发一张图，让 AI 把它变得好玩，让 NPC 捧场」的 AI 原生社交产品——把任意日常瞬间（不论倒霉还是幸福），重构成有人懂、有人笑、有人评论的 Drama 现场。**

它做三件事：

- **看懂你的无聊日常**
- **用戏剧化、有反差感的 AI 视觉手法重构现实**（生成有梗的图片/动图）  
- **一键发布并提供 AI 互动（点赞 + 评论）**的情绪价值闭环（25 秒内陆续出现）

### 1.1 评委偏好维度（强约束：需求与验收口径）

> 本小节同步了腾讯官方评委曝出的喜爱维度。后续新增功能、改交互、改模型策略、改 prompt/模板，都必须逐条对齐；若不满足，应视为“不通过验收”。

1) **真实痛点（避免伪需求）**  
从用户角度出发，优先解决“分享焦虑 / 情绪宣泄 / 发布后社交空窗期”等高频真实问题；避免为了展示能力而增加无意义步骤或功能堆叠。

2) **可真实落地（避免天马行空）**  
所有核心链路应可在现有工程形态（Serverless + Supabase + TokenHub）下闭环跑通：可部署、可调用、可存储、可回放、可复现。对外部依赖（模型/存储/带宽/权限）要有降级与失败反馈。

3) **AI 原生（拿掉 AI 就难以正常使用）**  
产品价值必须由 AI 核心能力驱动：多模态感知（看懂图）、决策收敛（Guess&Refine）、可控生成（结构一致性重绘）、拟真互动（NPC 评论/点赞）。若移除 AI，这条链路应无法以传统手段等价实现。

4) **避免技术自嗨**  
技术选择与模型参数应服务体验，不以“更复杂/更先进”为目标；任何新增的模型、链路、参数、组件，都必须能明确带来用户可感知的收益（更快、更稳、更像真人、更好玩）。

5) **趣味性与创新性兼具**  
趣味来自“现实瞬间 → 戏剧化重构 → 社交反馈”的反差与共鸣；创新来自“免 prompt 的 Guess&Refine + 结构一致性重绘 + NPC 社交闭环”的组合，而不是单点炫技。

6) **赛题④难点：风格控制与去 AI 感（提示词/策略）**  
需要稳定控制图片/视频/链接等输入输出的风格一致性，避免“生硬的 AI 味”。策略方向包括但不限于：  
- 通过结构约束（参考图/边缘/景深等）保持空间一致性  
- 通过系统级 Prompt 模板与负向约束保持风格稳定；Drama 元素须同时满足「视觉因果」（从图像锚点生发）与「情绪因果」（是场景情绪的视觉具现，禁止与情绪叙事无关的随机物体插入），评判标准为「意料之外、情理之中」  
- 通过“可选项按钮”替代自由输入 prompt，降低语义漂移与随机性  
- 通过语言与互动人设约束，使文案更像真人表达（克制、不油、不夸夸群）

### 2. 核心功能（解决核心痛点的抓手）

- **万物皆可 Drama（多模态感知输入）**  
  支持用户发送任意场景图片/短视频，AI 提取实体、场景关系与用户潜在情绪状态。

- **Guess & Refine 情绪互动罗盘**  
  弱化「从零写长 Prompt」：由视觉理解结果驱动，语言模型生成一句共情破冰文案，并给出 3 个可点的 Drama 方向供一键选择，降低意图表达门槛。每个改造方向须同时满足「视觉因果」（Drama 元素从画面已有锚点自然生发）与「情绪因果」（Drama 元素是当前情绪的视觉具现，而非随机插入），达到「意料之外、情理之中」的惊喜感，而非荒诞无厘头。风格示例以产品内实际配置为准，避免为炫技堆砌猎奇词。

- **空间一致性重绘（参考图 + 语义约束）**  
  以原图作为参考输入，并在生成指令中强调锁定关键物理元素与透视；前端可展示原图边缘骨架作为「结构锁定」的可视化提示（与 README 中「不等同于服务端 ControlNet 管线」的口径一致）。目标是戏剧化替换材质与氛围的同时，尽量保持「仍能看出是那张日常图」的传播性。

- **AI 赛博社交场（NPC Agent）**  
  一键分享至内部 Drama 广场。发布后，5 位 AI 互动角色（Emma、Liam、Olivia、Noah、Sophia）会在 **25 秒内陆续点赞与评论**，形成“电子情绪价值反馈”，消灭发布后的社交空窗焦虑。  
  - **风格要求**：更像真实的人类表达，不过度浮夸；每位角色有稳定但克制的表达偏好（语气、关注点、用词习惯不同）。

### 3. 产品架构与逻辑流程图（感知-决策-生成 任务拆解）

> 注：在 PDF 中可配对应思维导图/流程图。此处给出逻辑描述，直接回应赛题 3 和赛题 4。

整体产品链路以四个 AI 模块接力完成：**感知（YT-VITA）- 决策（HY-3 Preview）- 生成（HY-Image / HY-Video）- 反馈（NPC Engine）**。其中“感知-决策-生成”直接回应赛题要求，“反馈”负责发布后的 AI 社交闭环。

#### 环节一：用户发图与【感知层】（意图拆解）

- 用户上传真实生活照片（可选一句话吐槽）
- 前端经服务端路由将图像提交至 **腾讯 TokenHub** 上的多模态视觉模型（如 `youtu-vita`，以环境变量为准），返回结构化理解结果，例如：

```json
{
  "核心实体": "电脑屏幕及密集文字",
  "环境": "昏暗书桌",
  "情感标签": "焦虑/高压"
}
```

#### 环节二：AI 互动与【决策层】（Guess & Refine）

- 中枢调度 Agent 接收情感标签，经 **HY-3 Preview** 文本模型生成破冰文案、Drama 方向与可执行生成提示词
- 界面弹出破冰回复，例如：“被DDL冲烂了吧哥们？选个姿势发疯：”
- 并发呈现 3 个风格按钮（预埋不同系统级 Prompt 模板），用户点击完成意图收敛与确认

#### 环节三：AI 回图与【生成分发层】（空间控制与渲染）

- 前端可从原图提取边缘图（Canvas + Canny）用于 UI 上的「结构锁定」提示；生成侧以参考图 URL 与叙事体 Prompt 调用 **TokenHub** 上的图像/视频任务模型（如 `hy-image-v3.0` / `hy-video-1.5`，以环境变量与控制台开通为准）
- 回传最终图片或视频，用户可一键发布至广场
- 服务端触发 NPC 点赞与评论生成：根据生成结果与人设生成符合角色的互动内容，完成情绪闭环

### 4. 创新与差异化

相较于美图秀秀/妙鸭相机等偏写真形象定制产品，以及 Midjourney 等通用生图工具：

- **弱化长 Prompt：「先共情、再选项」的交互路径**  
  许多工具仍以长文本 Prompt 为主路径。小题大Drama 将主路径设计为「视觉理解 → 一句破冰 → 三个可点方向 → 参考约束生成」，降低从零描述意图的成本。业界亦有模板化、助手式或选项式生图产品；本产品的表述重点在于上述链路与本仓库实现一致、可演示，而非宣称「全球/国内唯一」。

- **结构锁定：反差建立在可感知的「仍是原图」之上**  
  写真类产品多聚焦人像与模板；Midjourney 等工具亦提供 `--cref`/`--sref` 等参考能力，熟练用户可做强一致性控制。小题大Drama 面向「随手日常图」场景，以参考图 + 语义锚定指令为默认路径，并在 UI 上提示边缘骨架，强调普通用户无需自行调参即可获得可预期的结构保留感。

- **NPC 社交闭环：从出图延伸到即时反馈**  
  许多生图产品止于导出文件。小题大Drama 在发布后为多名 NPC 人设触发点赞与评论（目标 ≤25s 内陆续出现），并可多轮回复。市面上亦有 AI 社交、游戏 NPC 等形态；本产品将「感知 → 选项化决策 → 参考约束生成 → 拟真互动」串成**同一 Web 产品内的闭环 Demo**，作为参赛作品的差异化叙事，不对「竞品数量」作全称判断。

## UI/UX 视觉与交互规范（Apple Style · 偏 macOS）

> 本章节是**后续所有前端迭代的强制设计约束**。任何新增页面/组件/动效，必须以此为准，确保风格一贯。

### 1. 设计哲学（必须遵守）

- **内容优先**：隐去工具痕迹，只留内容本身。避免“炫技式视觉”，信息层级清晰、可读性第一。
- **极致克制**：不用霓虹、发光、强渐变、赛博朋克未来风；避免高饱和+高对比造成的视觉疲劳。
- **精密秩序**：通过网格、留白、字体层级与细分隔线建立秩序，而非依赖装饰性效果。
- **材质呼吸感**：用轻微层级、细腻阴影、柔和灰阶表达“材质”，动效短促、安静、不抢内容。

### 2. 主题策略（必须遵守）

- **默认浅色为主**：全程以浅色体验为主，不允许因业务流程（如 Guess 成功）自动切换主题。
- **暗色模式可选**：仅允许“跟随系统”或“用户手动切换”，暗色风格为 **macOS 深色**（低饱和、低对比、无霓虹）。

### 3. 颜色体系（强约束）

- **唯一高亮色：系统蓝（Apple Blue）**  
  用途：主 CTA、可点击文本、聚焦态（focus ring）、选中态提示。  
  禁止：紫粉霓虹、发光描边、渐变背景作为主按钮。
- **灰阶为主**：背景/卡片/文字/分隔线均应使用中性灰阶，避免纯黑纯白硬对比。
- **语义色**：成功/失败/警告仅用于状态反馈（如任务完成/失败），面积要小、对比要克制。

### 4. 排版与信息层级（强约束）

- **标题**：字重偏中（semibold），避免过粗“海报感”。行高紧凑但不压迫，确保可读。
- **正文**：以清晰为先，减少花哨修饰；辅助信息使用更浅灰与更小字号。
- **数字/模型/ID**：使用等宽字体（mono），并避免过度强调。

### 5. 间距与布局（强约束）

- **留白是结构的一部分**：通过 padding/margin 建立节奏，不用装饰性线条堆砌。
- **不锁死最大宽度**：页面可随屏幕变宽而扩展，但**阅读内容区域**应保持合理行宽与栅格秩序（避免正文无限拉长）。
- **对齐与网格**：所有组件在同一列宽、同一基线对齐；边距要一致，不出现“随手堆叠”的凌乱感。

### 6. 组件外观规范（强约束）

- **非卡片化优先（强约束）**：本项目默认采用「朋友圈式信息流」的**平铺内容**表达方式，通过**留白 + 分隔线 + 字体层级**建立秩序，避免用大块 Card/Panel 把内容“框起来”。  
  - **允许的容器场景（例外）**：仅在“设置项分组 / 表单区块 / 强状态提示（错误/空态）/ 图片上传区”这类需要明确边界的区域，才允许使用轻量容器。  
  - **容器形态约束**：即使使用容器，也必须极轻（细边/弱底/无重阴影），不可出现“厚卡片感”和强分层。
- **按钮**：\n+  - 主按钮：系统蓝底白字，圆角中等（偏 macOS 面板），无渐变无发光。\n+  - 次按钮：浅灰背景/细边，hover 仅做轻微色阶变化。\n+- **选项选择（Option/Segmented）**：选中态使用浅蓝背景 + 细边或指示点；不要霓虹描边。
- **输入/上传区域**：强调“清晰可理解”，状态反馈（hover/dragging/error）差异要细腻但明确。

### 7. 动效与反馈（强约束）

- **动效短促**：以 150–250ms 的淡入/位移为主，避免弹跳/强 spring 影响克制感。
- **不做“赛博”特效**：禁止发光、粒子、霓虹闪烁、过度 blur 毛玻璃作为主视觉。
- **加载与等待**：信息要明确但不喧闹。允许极简呼吸点/文字渐隐渐显，避免大面积动画。

### 8. 禁止清单（直接拒绝）

- 霓虹渐变文字、霓虹发光阴影、赛博紫粉渐变 CTA
- 以“黑底+霓虹”为主导的视觉叙事（除非明确改 PRD 并说明原因）
- 过度毛玻璃（大面积 blur 覆盖内容导致可读性下降）
- 夸张动效（弹跳、强缩放、闪烁、复杂粒子）

## 模块三：AI 原生能力说明

### 1. AI 核心能力应用

小题大Drama 是一个 AI-Native 产品，深度融合四个 AI 模块：

- **多模态理解（Vision LLM）**：产品“眼睛”，负责图像语义拆解与情绪识别，支撑后续选项与文案  
- **HY-3 Preview 决策模型（Text LLM）**：产品“大脑”，负责 Guess & Refine 的破冰文案、选项推荐、提示词收敛与必要的文本兜底  
- **可控生成（TokenHub 图像/视频任务）**：产品“画笔”，在参考图与指令约束下完成异步生成；前端边缘提取仅作 UI 提示，不等同于自建 ControlNet 推理管线  
- **NPC Engine（LLM & Agent）**：产品“嘴巴和捧场团”，负责 5 位角色评论、回复与拟真互动  

### 2. AI 如何解决痛点（去掉AI产品还能成立吗？）

**结论：去掉 AI，本产品将彻底崩溃，无法成立。**

- **缺乏 AI 视觉理解**：仅靠硬编码规则难以稳定从日常照片中推断场景与情绪，难以支撑「共情 + 多方向」的选项生成  
- **缺乏 AI 可控生成**：常规滤镜难以在保留主体透视的前提下完成语义级场景重绘；生成模型与参考约束是产品主价值的技术前提  
- **缺乏 LLM 驱动的 Agent**：固定话术无法针对每张不同生成图做拟人化、差异化的评论与多轮回复；NPC 互动依赖语言模型  

### 3. AI 技术方案（应对赛题 2 的挑战）

为保证极速响应与一致体验，采用前后端分离与云原生 Serverless：

- **前端交互与部署**：Next.js 应用托管于 **Vercel**，API Route 承担需保密的上游调用（与仓库当前实现一致）  
- **中枢逻辑与感知决策**：经 **TokenHub** 调用腾讯侧多模态与文本模型（具体模型名以环境变量与控制台为准）  
  - 多模态：`youtu-vita` 做原图分析 + 情绪等结构化理解  
  - 文本能力：`hy3-preview` 驱动 Guess & Refine 选项、破冰文案、提示词收敛与 NPC 评论等  
- **生图与一致性保障**  
  - **结构一致性**：参考图传入 + 指令中锁定关键物理元素；前端 Canny 边缘图用于 UI 提示  
  - **风格一致性**：系统级 Prompt 模板与产品内风格策略（避免为文案堆砌不可验证的「引擎」名称）  
  - **任务形态**：图像/视频异步任务提交与轮询查询，耗时受上游队列与网络影响，以线上实测为准  

#### 部署与落地架构（Vercel + Supabase，无传统后端）

本项目目标是做到“拿到仓库即可部署”，因此采用 **全托管 + Serverless** 的工程形态：不运行常驻后端进程、不自建服务器，以 Next.js API Routes（Vercel Serverless）与托管数据库/存储完成业务闭环（与当前仓库 README 一致）。

- **Vercel（应用托管 + Serverless Functions）**  
  - 托管 Next.js 前端与路由  
  - 承载需要保密的服务端逻辑（如：调用大模型/生图接口的密钥、签名、风控校验、请求聚合）

- **Supabase（数据库 + Auth + Storage）**  
  - **Postgres**：存储用户、作品、发布、互动等结构化数据  
  - **Auth**：邮箱/手机号/OAuth 登录与会话管理（前端可直接使用 Supabase 客户端）  
  - **Storage**：存储用户上传原图/生成结果图/录屏等文件，并通过策略控制访问权限

#### 典型请求流（从“发图”到“回图”）

1. 前端在 Vercel 上提供上传与选择风格的交互  
2. 原图可直传 Supabase Storage（或经函数中转），并写入 Postgres 记录元数据  
3. 前端调用 Vercel Serverless Function：函数读取必要信息，安全地调用多模态/生图 API  
4. 生成结果落 Supabase Storage，并回写生成记录、评论/互动记录到 Postgres  
5. 前端拉取结果与互动数据，完成广场浏览与互动展示

> 说明：在“无传统后端”的约束下，所有需要私密凭证的第三方调用，都应放在 Vercel 的 Serverless Functions（Next.js Route Handlers）中完成；前端仅拿到受控的短期凭证/公开资源地址，避免密钥泄露。若团队另行接入 CDN/边缘网络（如 Cloudflare），属部署增强项，非本仓库默认依赖。
